# Plan: Entity Relationships, Field Classification, and Add/Delete Functionality

## Executive Summary

Based on deep analysis of live BeProduct JSON data (13 styles, 15 materials, 6 colors, 4 images, 5 blocks, 5 directory records, 14 users) and the full BeProduct SDK v0.6.30 API surface, this plan documents:

1. **Verified cross-entity relationships** (foreign keys between JSON domains)
2. **Entity creation sequence** (dependency order)
3. **Essential fields per entity** (primary keys / required fields)
4. **Read-only vs editable field classification**
5. **Add / delete functionality** — what the SDK supports and how to implement it

## 1. Cross-Entity Relationship Map (Verified from Live Data)

```
                    ┌──────────┐
                    │  Users   │
                    └────┬─────┘
                         │ createdBy.id / modifiedBy.id
                         │ (all entities reference users)
                         ▼
    ┌─────────────────────────────────────────────────┐
    │          Fields of type "Users"                 │
    │  (tech_designer, designer on Style/Material)    │
    └─────────────────────────────────────────────────┘

    ┌────────────┐         ┌───────────┐
    │ Directory  │◄────────│  Style    │  via PartnerDropDown.code
    │ (partners) │◄────────│ Material  │  via PartnerDropDown.code
    └────────────┘         └─────┬─────┘
                                 │
    ┌────────────┐               │ colorway.colorSourceId
    │  Color     │◄──────────────┤ (maps to color palette's
    │ (palette)  │◄──────────┐   │  headerData.colors.colors[].color_source_id)
    └────────────┘           │   │
                             │   │ colorway.imageHeaderId
    ┌────────────┐           │   │ (maps to images.id)
    │  Image     │◄──────────┼───┘
    └────────────┘           │
                             │
    ┌────────────┐           │ colorway.colorSourceId / imageHeaderId
    │ Material   │───────────┘
    └─────┬──────┘
          │
    ┌─────┴─────┐
    │  Block    │  No direct FK to Material found in data.
    └───────────┘  Block ↔ Material link is INDIRECT:
                   - Style field "reference_style_no" (BLOCK REFERENCE) is a text field
                   - Blocks define size classes that Styles inherit
                   - No enforced FK in JSON — operational convention only
```

### Confirmed References (with evidence)

| Source Entity | Source Field | Target Entity | Target Field | Evidence |
|---------------|-------------|---------------|--------------|----------|
| **Style/Material** colorway | `colorSourceId` (UUID) | **Color** palette | `headerData.colors.colors[].color_source_id` | 12/15 matched in live data. Same color reused across Style AND Material |
| **Style/Material** colorway | `imageHeaderId` (UUID) | **Image** | `id` (primary key) | Material colorway `0e9dd1ab...` found in images table |
| **All entities** | `createdBy.id`, `modifiedBy.id` | **Users** | `id` | 10/10 style createdBy IDs found in users table |
| **Style/Material** fields | `PartnerDropDown` field `.code` (UUID) | **Directory** | `id` | Style factory field code `00e75212...` = Directory "BeProduct Vendor" |
| **Style/Material** fields | `Users` field type | **Users** | User lookup | `tech_designer`, `designer` fields reference user records |

### Deep Investigation: Block ↔ Material ↔ Style (API-verified)

**Finding: No Block ↔ Material direct FK exists.** The implementation team's observation
is about an *operational workflow*, not a data-level foreign key.

The actual structural bonds discovered:

#### Block → Style: Shared sizeClass UUIDs (CONFIRMED)

When a Style is created referencing a Block, it **inherits the Block's sizeClass definitions
by UUID**. Out of all sizeClasses in the dataset:

| Relationship | Count |
|---|---|
| Shared sizeClass UUIDs (same ID in both Block and Style) | **6** |
| Block-only sizeClass UUIDs | 5 |
| Style-only sizeClass UUIDs | 7 |

Example of a shared sizeClass (`XS-XXL (S)`):
- Block `Otter Avenue` has sizeClass `db43b95e...` with sizes: XS, S, M, L, XL, XXL
- Style records also have sizeClass `db43b95e...` with **identical** sizes

This is the mechanism: Blocks are **size templates**. They define sizeClasses that Styles
consume by reference (same UUID). The sizes array is inherited verbatim.

#### Style → Block: Free-text BLOCK REFERENCE field

Style field `reference_style_no` (UI label: "BLOCK REFERENCE") is type `Text`, storing
a human-readable block name/number — NOT a UUID. Examples from live data:
- Style `LF123` → reference_style_no = `"5342"` (not a block UUID)
- Style `Womens SS Button Down` → reference_style_no = `"Walmart 2026"` (matches block headerNumber)

This is a **manual/display-only** reference. No API enforcement.

#### Block → Material: NO relationship found

- 0 out of 15 materials reference "block" anywhere in their JSON
- 0 out of 5 blocks reference "material" or "style" in their JSON
- Block fields contain only: `header_number`, `header_name`, `created_by`, `modified_by`,
  `active`, `PageModified`, `notes`, `brands_multi`, `sold_to_customer`,
  `product_category_multi`, `gender_multi`, `team`, `department_no`
- No `colorways`, no `suppliers`, no material links of any kind

#### Summary of Operational Workflow vs Data Structure

The implementation team's perceived sequence (user → folder → block → material → style)
reflects the **product development workflow** in the portal UI:

```
1. Admin creates Users                (to assign designers, tech designers)
2. Admin creates Directory partners   (factories, vendors)
3. Designer creates Block             (size template for a product category)
4. Sourcing creates Material          (fabric/trim with colorways)
5. Designer creates Style             (references Block for sizing, Material via BOM app)
```

But the **API does NOT enforce** this order. All entities can be created independently.
The cross-references (colorSourceId, imageHeaderId, PartnerDropDown.code, Users fields)
are resolved at display time and will simply show empty/broken if the target doesn't exist.

### NOT Confirmed (Indirect / Operational Only)

| Hypothesized Link | Finding |
|-------------------|---------|
| **Block → Material** | No FK found in Block JSON. Block defines size classes only. Blocks have no `colorways` or `suppliers`. The association to Material is only through BOM apps at the Style level, not in the attributes JSON. |
| **Style → Block** | Style field `reference_style_no` (type `Text`, labeled "BLOCK REFERENCE") is a plain text value, not a UUID FK. The real structural bond is **shared sizeClass UUIDs** — Styles inherit Block sizeClasses by reference, but this is a copy-on-create relationship, not an ongoing FK. |
| **Material → Block** | Zero references to blocks found in any of the 15 material records. |

## 2. Entity Creation Sequence (Dependency Order)

Based on the relationship map, the logical creation order is:

```
Level 0 (no dependencies):
  Users             — standalone; needed for createdBy/modifiedBy audit trail
  Directory         — standalone; needed for PartnerDropDown fields

Level 1 (depends on Level 0):
  Color Palettes    — standalone data; referenced by Style/Material colorways
  Images            — standalone data; referenced by Style/Material colorways
  Blocks            — standalone data; provides size class templates

Level 2 (depends on Level 0 + Level 1):
  Materials         — references Color (via colorway.colorSourceId),
                      Image (via colorway.imageHeaderId),
                      Directory (via PartnerDropDown fields),
                      Users (via Users-type fields)

Level 3 (depends on all above):
  Styles            — references Color, Image, Directory, Users,
                      and conceptually Block (via BLOCK REFERENCE text field)
```

**Important**: This is the *logical* order for data entry in the product portal. The API does NOT enforce foreign key constraints — you CAN create a Style without first creating its referenced Colors/Images. The references are resolved at display time. However, **creating in the correct order ensures data integrity**.

## 3. Essential Fields Per Entity (Primary Keys & Required Fields)

### Field Metadata from API

Every field object in `headerData.fields[]` has exactly 5 properties:
```json
{ "id": "...", "name": "...", "type": "...", "value": ..., "required": true/false }
```

The `required` flag is set per-field in the folder schema. There is NO `readOnly` property in the field data itself — read-only behavior is determined by field **type**.

### Primary Identifiers

| Entity | Primary Key | Display Key | Notes |
|--------|-------------|-------------|-------|
| **Style** | `id` (UUID, server-generated) | `headerNumber` | `id` is assigned by BeProduct on create |
| **Material** | `id` (UUID, server-generated) | `headerNumber` | Same |
| **Color** | `id` (UUID, server-generated) | `colorPaletteNumber` | Note: NOT `headerNumber` |
| **Image** | `id` (UUID, server-generated) | `headerNumber` | Same |
| **Block** | `id` (UUID, server-generated) | `headerNumber` | Same |
| **Directory** | `id` (UUID, server-generated) | `directoryId` + `name` | `directoryId` is user-assigned |
| **User** | `id` (UUID, server-generated) | `email` / `username` | Unique by email |

### Required Fields for Creation

Based on SDK method signatures and the `required` field property:

| Entity | Required for `attributes_create()` | Optional |
|--------|-------------------------------------|----------|
| **Style** | `folder_id`, `fields` (dict of field_id→value; at minimum `header_number`, `header_name`) | `colorways`, `sizes`, `force_version_update` |
| **Material** | `folder_id`, `fields` (same pattern) | `colorways`, `sizes`, `suppliers`, `force_version_update` |
| **Color** | `folder_id`, `fields` (at minimum `header_number`, `header_name`) | `colors`, `force_version_update` |
| **Image** | `folder_id`, `fields` | `force_version_update` |
| **Block** | `folder_id`, `fields` | `size_classes` |
| **Directory** | `fields` dict: `name`, `directoryId`, `partnerType` | `address`, `country`, `city`, `state`, `zip`, `phone`, `website` |
| **User** | `fields` dict: `email`, `username`, `firstName`, `lastName` | `title`, `accountType`, `role` |

| Field Type | Read-Only? | Behavior | Examples |
|------------|-----------|----------|----------|

| `Label3dMaterial` | **Yes** | Auto-populated from 3D linking | `core_3d_material` |
| `Text` | No | Free text input | `header_number`, `header_name`, `notes` |
| `Memo` | No | Multi-line text | `notes` |
| `DropDown` | No | Single selection from predefined list | `season`, `year`, `style_status` |
| `MultiSelect` | No | Multiple selections | `brands_multi`, `sold_to_customer` |
| `ComboBox` | No | Dropdown with free text entry | |
| `TrueFalse` | No | Boolean toggle | `active` |
| `PartnerDropDown` | No | Directory partner selection | `factory`, `parent_vendor`, `mill_vendor` |
| `Users` | No | User selection | `tech_designer`, `designer` |
| `Measure` | No | Numeric with unit | |
| `Currency` | No | Money value | |
| `Decimal` / `Number` | No | Numeric | |
| `Percent` | No | Percentage | |
| `Weight` | No | Weight value | |
| `Date` / `DateTime` | No | Date picker | |
| `CompositeControl` | No | Structured input (e.g., fabric content) | `fabric_content` |
| `FormulaField` | **Yes** | Calculated formula | |

### System-Level Read-Only Properties

These top-level JSON properties are **always** read-only (server-managed):

| Property | Read-Only? | Notes |
|----------|-----------|-------|
| `id` | **Yes** | Server-generated UUID. Cannot be changed. |
| `createdBy` | **Yes** | Auto-set on creation |
| `createdAt` | **Yes** | Auto-set on creation |
| `modifiedBy` | **Yes** | Auto-set on update |
| `modifiedAt` | **Yes** | Auto-set on update |
| `isDeleted` | **Yes** | Managed by server on delete |
| `folder` | **Partially** | Folder assignment is set at creation; moving items between folders is separate |
| `colorways[].id` | **Yes** | Server-generated colorway UUID |
| `colorways[].image` | **Yes** | Server-managed (set via upload) |
| `sizeRange[].sizeIndex` | **Yes** | Server-assigned ordering |

### Editor Field Behavior Summary

```
DISABLED (grey, non-editable):
  - id (hidden or displayed as read-only label)
  - createdBy, modifiedBy (UserLabel type)
  - createdAt, modifiedAt (system timestamps)
  - All Label* types (LabelText, LabelMaterial, LabelSize, etc.)
  - FormulaField
  - isDeleted

EDITABLE:
  - header_number / header_name (Text — the "display name")
  - All DropDown, MultiSelect, ComboBox fields
  - All Text, Memo fields
  - TrueFalse (active)
  - PartnerDropDown (links to Directory)
  - Users (links to Users)
  - Measure, Currency, Number, Decimal, Percent, Weight, Date/DateTime
  - CompositeControl (fabric content)
```

## 6. Schema-Based Field Editor (SDK Integration)

### How the SDK Schema Module Works

The `client.schema.get_folder_schema(master_folder, folder_id)` method calls:
```
GET {master_folder}/FolderSchema?folderId={folder_id}
```

It returns a processed list of field metadata dicts:
```python
{
    "field_id": str,          # e.g. "season", "year", "factory"
    "field_name": str,        # e.g. "SEASON", "YEAR", "FACTORY"
    "field_type": str,        # e.g. "DropDown", "MultiSelect", "PartnerDropDown"
    "required": bool,         # True if field is required for creation
    "formula": str,           # populated for FormulaField / LabelText types
    "possible_values": list,  # populated for DropDown/MultiSelect/ComboBox/PartnerDropDown
    "data_type": type,        # Python type: str, bool, datetime, float, int, list
    "properties": dict,       # raw field properties from API
}
```

For `DropDown`/`MultiSelect`/`ComboBox`, `possible_values` is a list of dicts:
```python
[{"id": "Spring", "code": "Spring", "value": "Spring"}, ...]
```

For `PartnerDropDown`, `possible_values` lists directory partners:
```python
[{"id": "00e75212-...", "code": "00e75212-...", "value": "BeProduct Vendor"}, ...]
```

Convenience methods: `get_style_folder_schema(folder_id)`,
`get_material_folder_schema(folder_id)`, `get_color_folder_schema(folder_id)`.
Note: **no `get_block_folder_schema`** or **`get_image_folder_schema`** convenience —
use `get_folder_schema("Block", folder_id)` / `get_folder_schema("Image", folder_id)`.

### Schema Caching Strategy

Folder schemas rarely change. Cache per `(master_folder, folder_id)` key in
`st.session_state` with a 10-minute TTL to avoid repeated API calls:

```python
@st.cache_data(ttl=600)
def get_cached_folder_schema(master_folder: str, folder_id: str) -> list[dict]:
    client = get_client()
    return client.schema.get_folder_schema(master_folder, folder_id)
```

### Folder Data Already Cached in SQLite

All five entity tables already store `folder_id` and `folder_name`. The locally known
folders (from synced records) can serve as fallback when SDK `folders()` call isn't
feasible. For the "Create New" dialog, calling `client.{entity}.folders()` is needed
to show all available folders (some may have zero synced records locally).

## 7. User Decisions (Confirmed)

| Decision | Choice |
|----------|--------|
| Block ↔ Material relationship | Investigated via API. **No direct FK exists.** Block→Style link is via shared sizeClass UUIDs. Block→Material link is purely operational (through BOM apps at Style level). |
| Add / Delete scope | **Full CRUD**: Create + Delete for Style/Material/Color/Image/Block. Create-only for Directory. Create + Edit for User. No delete for Directory or User (SDK limitation). |
| Field editor approach | **Schema-based**: Fetch folder schema from SDK (`get_folder_schema`) to populate DropDown options, field validation rules, and required flags. Cache with 10-minute TTL. |

## 10. Data Tables — New Entity Discovery

### Finding

The BeProduct portal sidebar shows "Data Tables" as a separate module. These are **custom
lookup/mapping tables** defined by the company admin (e.g., "E1 Color Codes" with columns
"E1 COLOR CODE", "EL COLOR NAME", "CREATED", "MODIFIED").

### SDK Status

The Python SDK v0.6.30 does **NOT** wrap DataTable endpoints — there is no `client.datatable`
handler. However, the **REST API does expose full CRUD** for Data Tables, confirmed from
the live Swagger spec at `https://developers.beproduct.com/swagger/v1/swagger.json`.

### REST API Endpoints (from OpenAPI spec)

| Method | Endpoint | Summary |
|--------|----------|---------|
| POST | `/api/{company}/DataTable/List` | List all data tables (paginated, with search) |
| GET | `/api/{company}/DataTable/{dataTableId}/Schema` | Get data table column schema |
| POST | `/api/{company}/DataTable/{dataTableId}/Data` | Search/list rows in a data table |
| POST | `/api/{company}/DataTable/{dataTableId}/Update` | Add, update, or delete rows |
| POST | `/api/{company}/DataTable/{dataTableId}/Reset` | Reset a data table |

### Data Models (from OpenAPI spec)

**DataTableResult** (list item):
```json
{
  "id": "uuid",
  "name": "string",
  "description": "string",
  "active": true,
  "createdBy": {"id": "uuid", "name": "string"},
  "createdAt": "datetime",
  "modifiedBy": {"id": "uuid", "name": "string"},
  "modifiedAt": "datetime"
}
```

**DataTableRowResult** (a row of data):
```json
{
  "id": "uuid",
  "fields": [{"id": "field_id", "name": "Field Name", "type": "Text", "value": "..."}],
  "createdBy": {"id": "uuid", "name": "string"},
  "createdAt": "datetime",
  "modifiedBy": {"id": "uuid", "name": "string"},
  "modifiedAt": "datetime"
}
```

**DataTableRowUpdateRequest** (upsert/delete a row):
```json
{
  "rowId": "uuid | null",      // null = insert new row
  "rowFields": [{"id": "field_id", "value": "new_value"}],
  "deleteRow": false           // true = delete this row
}
```

**DataTableRowsUpdateResult** (response from update):
```json
{
  "updated": ["uuid", ...],   // IDs of updated rows
  "added": ["uuid", ...],     // IDs of newly inserted rows
  "deleted": ["uuid", ...]    // IDs of deleted rows
}
```

### Connection to Color Palette

Each individual color in a palette has a `Schema` field. Its keys directly correspond
to Data Table columns:
```json
"Schema": {"e1_color_code": null, "el_color_name": null}
```
The "E1 Color Codes" Data Table provides the lookup values that map to these schema
extension fields on colors. Currently all values are `null` (table is empty).

### Implementation Approach

Since the SDK doesn't wrap these endpoints, we use `client.raw_api` directly:

```python
from app.beproduct_client import get_client

client = get_client()
domain = settings.COMPANY_DOMAIN

# List all data tables
tables = client.raw_api.post(f"DataTable/List", body={})

# Get schema for a specific data table
schema = client.raw_api.get(f"DataTable/{table_id}/Schema")

# Get rows from a data table
rows = client.raw_api.post(f"DataTable/{table_id}/Data", body={})

# Update rows (add/edit/delete)
result = client.raw_api.post(f"DataTable/{table_id}/Update", body=[
    {"rowId": None, "rowFields": [{"id": "e1_color_code", "value": "001"}], "deleteRow": False},
])
```

### Data Table SQLite Schema

```sql
CREATE TABLE IF NOT EXISTS data_tables (
    id TEXT PRIMARY KEY,
    name TEXT,
    description TEXT,
    active INTEGER DEFAULT 1,
    created_at TEXT,
    modified_at TEXT,
    synced_at TEXT,
    data_json TEXT
);

CREATE TABLE IF NOT EXISTS data_table_rows (
    id TEXT PRIMARY KEY,
    data_table_id TEXT NOT NULL,
    created_at TEXT,
    modified_at TEXT,
    synced_at TEXT,
    is_dirty INTEGER DEFAULT 0,
    data_json TEXT,
    FOREIGN KEY (data_table_id) REFERENCES data_tables(id)
);
CREATE INDEX idx_dt_rows_table ON data_table_rows(data_table_id);
CREATE INDEX idx_dt_rows_dirty ON data_table_rows(is_dirty);
```

# Plan: Refresh & Validate Local DB Against Latest BeProduct Setup

**Objective**: Perform a clean sync of Walmart and KTB customer data, validate schema alignment, check data integrity, and investigate unmapped masters (BOM, Spec, Techpack, Sample).

**Date**: 2026-05-06  
**Scope**: Full DB refresh + schema validation + relationship integrity + investigation of missing masters

---

## Phase 1: Database Reset & Full Sync

### 1.1 Clean Database Reset
- **Action**: Delete existing `data/beproduct.db` (if it exists)
- **Reason**: Ensure clean state for full sync without stale data
- **Impact**: Irreversible — recommend archiving current DB first if needed

### 1.2 Initialize Fresh Schema
- **Action**: Call `db.init_schema()` to create all tables with current DDL
- **Scope**: 
  - Entity tables: `styles`, `materials`, `colors`, `images`, `blocks`
  - Master data: `users`, `directory`
  - Data tables: `data_tables`, `data_table_rows`
  - Metadata: `sync_meta`, `rate_limit_log`

### 1.3 Execute Full Sync
- **Action**: Run `sync.sync_all(force_full=True)` to fetch all records from BeProduct API
- **Scope per entity**:
  - **Styles**: All folders (including Walmart, KTB, others)
  - **Materials**: All folders
  - **Colors**: All palettes
  - **Images**: All images
  - **Blocks**: All blocks
  - **Users**: All users
  - **Directory**: All partner records
  - **Data Tables**: All custom lookup tables
- **Expected outcome**: Complete local copy of all Walmart and KTB data

### 1.4 Validate Sync Completion
- **Action**: Query row counts per entity
- **Success criteria**: 
  - All Walmart folder records present
  - All KTB folder records present
  - `synced_at` timestamp set for all records
  - No `is_dirty=1` records (all clean after sync)

## Phase 2: Schema Alignment Validation

### 2.1 Compare Local Schema vs API Response Shapes
- **Action**: Sample records from each entity and validate against documented shapes in KNOWLEDGE.md
- **Validation checks**:
  - **Styles**: Verify `headerData.fields[]` structure, `colorways[]`, `sizeRange[]` present
  - **Materials**: Same as styles
  - **Colors**: Verify `colorPaletteNumber`, `colorPaletteName` (not `headerNumber`/`headerName`)
  - **Images**: Verify basic structure
  - **Blocks**: Verify `sizeClasses[]` and `headerData.fields[]`
  - **Directory**: Verify `directoryId`, `partnerType`, `contacts[]` structure
  - **Users**: Verify `registerdOn` (typo in API) is correctly mapped
- **Output**: Schema validation report with any mismatches

### 2.2 Validate Field Type Mappings
- **Action**: Check actual field types in `headerData.fields[]` match expected patterns
- **Scope**: Sample 10-20 records per entity from Walmart folder
- **Validation**: Confirm known read-only types (`UserLabel`, `LabelText`, `FormulaField`, `Auto`) vs editable types
- **Output**: Field type coverage report

### 2.3 Verify Data Table Integration
- **Action**: List all data tables and their schemas
- **Check**: Verify color palette `Schema` field references (e.g., `e1_color_code` maps to "E1 Color Codes" data table)
- **Output**: Data table dependency map

## Phase 3: Data Integrity Validation

### 3.1 Foreign Key Relationship Validation
- **Action**: Verify all documented relationships from KNOWLEDGE.md §6
- **Checks**:
  
  | Relationship | Validation |
  |---|---|
  | Style/Material `colorways[].colorSourceId` → Color `color_source_id` | Spot-check 20 colorways, verify each `colorSourceId` exists in color palette |
  | Style/Material `colorways[].imageHeaderId` → Image `id` | Verify image UUIDs exist in images table |
  | Style/Material `PartnerDropDown.value.code` → Directory `id` | Verify all factory/vendor UUIDs exist in directory |
  | Style/Material `Users` field → User `id` | Verify tech_designer, designer references exist |
  | All entities `createdBy.id` / `modifiedBy.id` → User `id` | Verify audit trail references |
  | Block `sizeClasses[].id` ← Style `sizeRange[].id` | Spot-check 10 styles, verify template inheritance |

- **Output**: Referential integrity report with any broken references

### 3.2 Cross-Folder Reference Check
- **Action**: Check if any Walmart records reference KTB records or vice versa
- **Scope**: Check colorways, partner links, image references between folders
- **Output**: Cross-folder dependency report

### 3.3 Orphan Record Detection
- **Action**: Find records with dangling references (e.g., styles referencing deleted colors)
- **Scope**: Run for Walmart and KTB separately
- **Output**: Orphan records report with impact assessment

### 3.4 Dirty Flag Validation
- **Action**: Verify no records have `is_dirty=1` after initial sync (should all be clean)
- **Expected**: All `is_dirty` values = 0
- **Output**: Dirty flag anomalies report

## Risks & Mitigations

| Risk | Mitigation |
|------|-----------|
| Data loss (deleting old DB) | Archive current `beproduct.db` before deletion |
| Incomplete sync (API throttling) | Built-in retry logic in `sync.py`; monitor rate limits |
| Orphan records from incomplete KTB | KTB status report will highlight these; mark for cleanup |
| Missing masters not discoverable via API | Will document in Phase 4.4; propose data-driven workarounds |
| Large dataset (Walmart + KTB combined) | Full sync typically completes in < 5 min; monitor API rate limits |
